Introduccin a la Computacin con GPUs usando R

1) Introduccin

Las GPU (Graphics Processing Units; Unidades de Procesamiento de Grficos) son unidades de procesamiento diseadas originalmente para procesar grficos en una computadora rpidamente. Esto se hace teniendo una gran cantidad de unidades de procesamiento simples para clculos masivamente paralelos. La idea de la computacin de propstio general en GPU (GPGPU: general purpose GPU computing) es explotar esta capacidad para el clculo general.

En este tutorial se revisar algunas formas de escribir clculos implementados en la GPU. El contexto bsico de la programacin de la GPU es el paralelismo de datos, en el que se realiza el mismo clculo para muchos datos. Esto podra ser un clculo matemtico sobre millones de entradas en un vector o una simulacin con muchas sub-simulaciones independientes. Algunos ejemplos de paralelismo de datos incluyen la multiplicacin de matrices (haciendo la tarea de multiplicacin en muchos elementos de matriz separados) o la integracin numrica (haciendo una estimacin numrica de la integral por partes en muchos intervalos/regiones), as como clculos estadsticos estndar como estudios de simulacin, bootstrapping, random forest, etc. Este tipo de clculo tambin se conoce con el nombre SIMD (single instruction, multiple data; instruccin nica, datos mltiples).

1.1) Hardware

Dos de los principales proveedores de dispositivos GPU son NVIDIA y AMD. CUDA es una plataforma para programar en GPUs, especficamente para GPUs NVIDIA, que permite enviar cdigo C/C++/Fortran para su ejecucin en la GPU. OpenCL es una alternativa que funciona con una variedad ms amplia de GPUs. De estos dos, CUDA ha alcanzado mayor popularidad y ltimamente est poniendo bastante nfasis en investigacin, especialmente en torno a tcnicas de deep learning y sus aplicaciones. Algunos ejemplos aqui.

Las GPUs tienen muchas unidades de procesamiento pero memoria algo limitada. Adems, solo pueden usar datos en su propia memoria, no en la memoria de la CPU, por lo que los datos debe ser transferidos entre la CPU (el host) y la GPU (el dispositivo). Esta copia puede, en algunos clculos, constituir una fraccin muy grande del clculo general. Por lo tanto, es mejor crear los datos y/o dejar los datos (para clculos posteriores) en la GPU cuando sea posible y disminuir al mximo las transferencias.

1.2) Librerias de R que utilizan computacin GPU

Estas son algunas de las libreras de R tiles para hacer clculos en la GPU desde R, mismas que se encuentran recopiladas en la seccin Computacin paralela: GPUs de la Vista de Tarea High Performance Computing.

  • gpuR: ofrece funciones habilitadas para GPU, nuevas clases gpu* y vcl* para envolver objetos tpicos de R (ej: vector, matriz) que reflejan la sintaxis R tpica sin la necesidad de conocer OpenCL.
  • cudaBayesreg: paquete que implementa el rhierLinearModel del paquete bayesm utilizando el lenguaje y las herramientas CUDA de nVidia para proporcionar un anlisis estadstico de alto rendimiento de voxels fMRI.
  • tensorflow: proporciona acceso a la API completa de TensorFlow desde R que permite el clculo numrico utilizando grficos de flujo de datos. La arquitectura flexible permite a los usuarios implementar clculos en una o ms CPU o GPU en un pc, servidor o dispositivo mvil con una sola API.
  • keras: interfaz hacia keras, una API de redes neuronales de alto nivel, desarrolada para permitir experimentacin rpida, con soporte para redes basadas en convolucin (convolution based networks ) y redes recurrentes (recurrent networks), y se ejecuta transparentemente tanto en dispositivos CPU como GPU.

3.2) Descripcin general de la computacin en una GPU

La secuencia bsica de operaciones para usar una GPU es:

  • asignar memoria en la GPU
  • transferir datos de la CPU a la GPU
  • lanzar el kernel CUDA para operar en los hilos, con una configuracin especfica de bloque/grid
  • (opcionalmente) ejecutar otro ncleo, que puede acceder a los datos almacenados en la GPU, incluidos los resultados del kernel anterior
  • transferir los resultados de regreso a la CPU

Los clculos clave se realizan en kernels, que son funciones que codifican las operaciones de cmputo centrales que se ejecutan en paralelo.

Como se indic anteriormente, debemos pasar cualquier dato de la CPU a la GPU y hacer lo mismo a la inversa para obtener el resultado. Tambin necesitaremos asignar memoria en la GPU. Sin embargo, en algunos casos, la transferencia y la asignacin se realizarn automticamente detrs de escena.

3.3) Capacidad de clculo CUDA

La capacidad de clculo (compute capability) se refiere bsicamente a la funcionalidad en evolucin de la arquitectura NVIDIA. Los nmeros ms altos proporcionan ms funcionalidad, pero solo se ejecutarn en hardware GPU ms nuevo.

Dos ejemplos:

  • Para usar dobles en lugar de flotantes, se necesita una capacidad de clculo de al menos 1.3.
  • Para usar la librera de Deep Learning de CUDA (cuDNN: NVIDIA CUDA Deep Neural Network library), se necesita una capacidad de clculo de al menos 3.0.

4.1) Operaciones de lgebra lineal

Comenzaremos con un uso de alto nivel de la GPU simplemente llamando a rutinas de lgebra lineal que usen la GPU. Dada la naturaleza numrica de la mayora de las aplicaciones R, es natural que existan paquetes desarrollados como interfaz entre R y GPU. Varios paquetes R ocultan los detalles de la implementacin de la GPU y permiten realizar operaciones vectoriales y matriciales, incluido el lgebra lineal, utilizando cdigo R estndar. En algunos casos, sobrecargan las funciones R habituales de modo que simplemente puede llamar a una funcin del mismo nombre que en la base R.

Algunos paquetes en este contexto incluyen: gpuR, gmatrix, gputools. Desafortunadamente, de estas 3 libreras, nicamente gpuR est disponible para la versin ms reciente de R, 3.5, liberada en abril de 2018.

gpuR proporciona funciones habilitadas para GPU para objetos R de una manera simple y accesible. Para acceder a estas funciones se proporcionan nuevas clases gpu* y vcl* para envolver objetos R tpicos (eg: vector, matriz), tanto en el host como en los espacios del dispositivo, para reflejar la sintaxis R tpica sin la necesidad de conocer OpenCL.

A continuacin, se hace una prueba simple en la que se compara el rendimiento de la multiplicacin de matrices sobre CPU versus GPU.

library(gpuR)
library(ggplot2)
library(profvis)

orders <- seq(1000,3000,by=1000)
#orders <- seq(1000,1600,by=200)
t1 <- t2 <- sizes <- c()

for(order in orders) {
  print(order)
  A = matrix(rnorm(order^2), nrow=order) 
  sizes <- c( sizes, format(object.size(A), units="auto") )
  gpuA = gpuMatrix(A, type="float")

  elapsed1 <- system.time({C = A %*% A})[3]  # Operacion regular (ejecutada en la CPU) 
  elapsed2 <- system.time({gpuC = gpuA %*% gpuA})[3] # Operacion acelerada (ejecutada en la GPU)

  t1 = c(t1, elapsed1)
  t2 = c(t2, elapsed2)
  
  rm(A,C,gpuA,gpuC)
  gc()
}

# Armar un dataframe con los resultados de las pruebas del for anterior
df = data.frame(order = rep(orders,2), 
                label = rep(paste0(orders," (", sizes, ")"), 2),
                mode = rep(c("cpu","gpu"),each=4), 
                elapsed = c(t1,t2))

# Guardar el dataframe (ya que el codigo de este chunk para plotearlo a continuacin)
# saveRDS(df, file = "data/gpuMm.rds")
library(ggplot2)
df <- readRDS(file = "data/gpuMm.rds")
ggplot(df, aes(x=label, y=elapsed, fill=mode)) +
  geom_bar(stat="identity", position=position_dodge()) +
  xlab("Orden (Mem)") + 
  ylab("Tiempo de ejecucion (seg)") #+ scale_y_log10()

4.2) Regresin Linear (OLS, Ordinary Least Squares)

En estadstica, los mnimos cuadrados ordinarios es el nombre de un mtodo para encontrar los parmetros poblacionales en un modelo de regresin lineal. Este mtodo minimiza la suma de las distancias verticales entre las respuestas observadas en la muestra y las respuestas del modelo. El parmetro resultante puede expresarse a travs de una frmula sencilla en notacin matricial:

\[ y = X \beta + \epsilon \] en donde \(y\) y \(\epsilon\) son vectores, y \(X\) es una matriz de regresores, a la que tambin se le llama la matriz de diseo.

La estimacin de la ecuacin anterior, se denota por:

\[ \hat{\beta} = (X'X)^{-1}X'y \] que puede ser re-escrita como: \[ (X'X) \hat{\beta} = X'y \] donde se debe resolver para \(\hat{\beta}\), que en manipulacin de matrices, resolver un sistema de ecuaciones es mucho ms rpido que invertir una matriz, y frecuentemente evita problemas numricos presentes alrededor de valores cercanos a cero.

set.seed(123456)
np <- 300  # nmero de predictores
nr <- 1e+05  # nmero de observaciones
X <- cbind(5, 1:nr, matrix(rnorm((np - 1) * nr, 0, 0.01), nrow = nr, ncol = (np - 1)))
beta <- matrix(c(1, 3, runif(np - 1, 0, 0.2)), ncol = 1)
y <- X %*% beta + matrix(rnorm(nr, 0, 1), nrow = nr, ncol = 1)

# Clculo en la CPU ligeramente optimizado via la utilizacin de producto cruzado 
timeCpu <- system.time({
  ms2 <- solve(crossprod(X), crossprod(X, y))
})

# Versin GPU, utilizando clases gpu* que proporcionan punteros de la GPUhacia la memoria de la CPU
library(gpuR)
gpuX = gpuMatrix(X, type = "float")  #point GPU to matrix
gpuy = gpuMatrix(y, type = "float")
timeGpu <- system.time({
  ms4 <- gpuR::solve(gpuR::crossprod(gpuX), gpuR::crossprod(gpuX, gpuy))
})

# Versin GPU (2), utilizando memoria interna del GPU!! (el llamado a vclMatrix es una transferencia)
vclX = vclMatrix(X, type = "float")  #push matrix to GPU
vcly = vclMatrix(y, type = "float")
timeGpu2 <- system.time({
  ms5 <- gpuR::solve(gpuR::crossprod(vclX), gpuR::crossprod(vclX, vcly))
})

detach("package:gpuR", unload = TRUE)

# Armar un dataframe con los resultados del bucle
df = data.frame(calculo = c("CPU", "GPU (gpu*)", "GPU (vcl*)"), 
                tiempo = c(timeCpu[3], timeGpu[3], timeGpu2[3]) )
# saveRDS(df, file = "data/gpuOLS.rds")
# Plotear los resultados

4.3) Ejemplo clsico de reconocimiento de imgenes MNIST

La base de datos MNIST (Modified National Institute of Standards and Technology database; Base de datos modificada del Instituto Nacional de Estndares y Tecnologa), es una gran base de datos de dgitos manuscritos que se usa comnmente para entrenar sistemas de procesamiento de imgenes. La base de datos tambin se usa ampliamente para entrenamiento y pruebas en el campo del aprendizaje automtico (machine learning).

Imgenes de muestra del conjunto de datos de prueba MNIST.

A continuacin se incluye el fragmento de un script que haciendo uso de las libreras keras y tensorflow, entrena un modelo de clasificacin de imgenes, que hace uso de las capacidades de la GPU. Hay que mencionar que estas libreras funcionan tanto sobre CPUs como GPUs. Para hacer uso de tensorflow con capacidades GPU, fue necesario instalar la librera NVIDIA CUDA Deep Neural Network (cuDNN).

library(keras)
use_condaenv("keras")
# install_keras(tensorflow="gpu") # Instalar solo la primera vez

# Preparando los datos
mnist <- dataset_mnist()
x_train <- mnist$train$x
y_train <- mnist$train$y
x_test <- mnist$test$x
y_test <- mnist$test$y

# Re-organizar
x_train <- array_reshape(x_train, c(nrow(x_train), 784))
x_test <- array_reshape(x_test, c(nrow(x_test), 784))
# Re-escalar
x_train <- x_train / 255
x_test <- x_test / 255


y_train <- to_categorical(y_train, 10)
y_test <- to_categorical(y_test, 10)

# Definir el modelo
model <- keras_model_sequential()
model %>%
  layer_dense(units = 256, activation = 'relu', input_shape = c(784)) %>%
  layer_dropout(rate = 0.4) %>%
  layer_dense(units = 128, activation = 'relu') %>%
  layer_dropout(rate = 0.3) %>%
  layer_dense(units = 10, activation = 'softmax')

summary(model)

model %>% compile(
  loss = 'categorical_crossentropy',
  optimizer = optimizer_rmsprop(),
  metrics = c('accuracy')
)

# Entrenar el modelo
history <- model %>% fit(
  x_train, y_train,
  epochs = 30, batch_size = 128,
  validation_split = 0.2
)

# Evaluar el modelo
model %>% evaluate(x_test, y_test)

# Predecir en datos nuevos
model %>% predict_classes(x_test)

5.1) Algunos pensamientos sobre la mejora de la velocidad computacional

Suchard et al (2010) y Lee et al (2010) hablan sobre el uso de GPU en estadstica. Las aceleraciones que reportan llegan a 120 y 500 veces la velocidad de un nico ncleo de CPU, respectivamente. Algunas de las razones por las cuales estas aceleraciones son tan impresionantes son:

  • Uso de clculos de precisin simple (single precision floating point calculations). Si la precisin no afecta sustancialmente el clculo, vale la pena intentarlo.

  • Tareas computacionales que son muy intensas en trminos aritmticos pero con acceso limitado a la memoria (ver el artculo de Lee et al.)

  • Garantizar que los hilos numerados contundentemente accedan a las ubicaciones de memoria contiguas. Ms informacin aqu.

  • Uso cuidadoso de la memoria compartida (compartida entre los hilos en un bloque) en lugar de la memoria principal de la GPU (ver el documento de Suchard et al.); en particular, esto puede evitar el acceso a la memoria no contigua.

  • Evitar declaraciones condicionales y sincronizacin/barreras, ya que los hilos operan en bloqueo-paso en grupos de 32 hilos (un warp)

Por lo tanto, para algunas tareas y probablemente con un esfuerzo adicional de codificacin, se puede conseguir aceleraciones de 100-200 veces en comparacin con un solo ncleo de CPU.
